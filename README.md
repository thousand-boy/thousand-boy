## 🎯 なぜ「現場で使われるAIアプリ」なのか

医療現場では、便利なはずのツールが「入力が面倒」「現場の流れに合わない」「結局使われない」という理由で定着しないのを何度も見てきました。**作る側ではなく使う側だったからこそ分かる「定着しない理由」** を、プロダクト設計の起点にしたいと考えています。

特に関心がある領域（いずれも「現場の手間を減らす」テーマ）：
- 社内・現場ドキュメントのRAG検索（必要な情報に最短で辿り着く）
- 議事録・記録の自動要約とタスク抽出
- 報告書・メールなどの下書き自動生成

## 🧭 学習・開発の方針

- **個人開発を主軸に**：動くアプリの「質」と、設計判断を語れる状態を優先する
- **品質ゲート制で進む**：日付ではなく「公開URLで動く／『課題→アプローチ→筋道』を3分で説明できる／READMEが第三者に伝わる／最小テストがある」を満たしてから応募へ
- **AIは段階的に活用**：基礎期は「答え合わせ」、開発期は「ペアプログラマ」として使い、読解力・設計力・エラーの切り分け力を自力で鍛える
- **テストはゼロにしない**：コア機能に最小限のテストとRAG評価を用意し、「保守を意識している」ことを示す
- **発信は任意**：設計判断や失敗からの改善を1〜2本記事化（言語化力の証明）
- **GitHubデイリーコミットを継続**：継続力・誠実さの可視化
